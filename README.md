
# 引擎電控單元（ECU）與 電子燃油噴射（EFI）編程入門

> 文件編制：2026 年 4 月  火箭鳥創客倉庫
> 
> 技術主題：引擎電控單元 · 電子燃油噴射 · 嵌入式系統開發

---

## 1. 簡介

電子燃油噴射（Electronic Fuel Injection, EFI）系統的核心在於**引擎電控單元**（Engine Control Unit, ECU）。ECU 如同引擎的大腦，負責即時接收各種感測器訊號，並透過複雜的演算法，精準控制噴油脈寬（Injector Pulse Width）與點火時機（Ignition Timing），以達到最佳的動力輸出、燃油效率及排放標準。

EFI 系統的出現取代了傳統化油器，使引擎在各種工況下均能維持準確的空燃比（Air-Fuel Ratio, AFR），大幅提升引擎性能與環保性能。

---

## 2. EFI 編程的核心邏輯

### 2.1 燃油方程式

ECU 的首要任務是根據**理想氣體狀態方程式**（Ideal Gas Law, PV = nRT）計算進入氣缸的空氣質量，進而決定噴油脈寬。核心運算公式如下：

```
IPW = K × (MAP × VE / T_air) × λ_target + Accel + Offset
```

| 參數 | 名稱 | 說明 |
|------|------|------|
| `K` | 常數 | 代表噴油嘴流量與排氣量的換算係數 |
| `MAP` | 進氣岐管絕對壓力 | 引擎負載的主要量測依據 |
| `VE` | 容積效率 | 引擎實際進氣量與理論進氣量之比，由 3D 查找表提供 |
| `T_air` | 進氣溫度 | 用於空氣密度修正（溫度越低，空氣越密，需要更多燃油） |
| `λ_target` | 目標空燃比 | 設定的理想燃油混合比例（巡航約 14.7:1，高負載約 12.5:1） |
| `Accel` | 加速補償 | 模擬化油器加速泵，節氣門快速開啟時額外補油 |
| `Offset` | 噴油嘴死區時間 | 補償噴油嘴的物理開啟延遲（通常為數微秒） |

### 2.2 核心控制查找表（Maps）

EFI 編程的核心之一是建立 **3D 查找表（Look-up Table）**，其 X 軸為引擎轉速（RPM），Y 軸為引擎負載（MAP 或 TPS）：

- **VE 容積效率表**：定義引擎在各轉速與負載組合下的實際進氣效率。
- **點火提前角度表（Ignition Advance Map）**：決定在上死點前（BTDC）幾度觸發火星塞。
- **目標空燃比表（Target AFR Map）**：設定各工況的理想燃油混合目標。

### 2.3 關鍵軔體例程

除基本供油外，固件還必須處理多項即時子程序：

- **中斷服務程序（ISR）**：由曲軸位置感測器（CKP）觸發的高優先級程式，確保 ECU 在毫秒甚至微秒級精準掌握活塞位置。
- **冷啟動加濃（Cold Start Enrichment）**：當冷卻水溫偏低時自動加大噴油量。
- **加速補油（Acceleration Enrichment）**：節氣門快速開啟時模擬傳統化油器的加速泵動作。
- **閉迴路修正（Closed-Loop Feedback）**：利用含氧感測器（O2 Sensor）偵測實際燃燒結果，即時修正噴油脈寬。

## 3. 硬體架構與 MCU 選用

### 3.1 車規級 MCU 需求

車用 ECU 對即時性（Real-time Determinism）與功能安全（Functional Safety）要求極高，通常需要：

- **工作溫度範圍**：−40°C 至 +125°C（部分應用更寬）
- **Lockstep 鎖步核心**：兩個處理器核心同時執行同一段程式，相互比對以即時偵測硬體錯誤
- **GTM（Generic Timer Module）**：專用計時子處理器，處理曲軸觸發輪的齒數計算與噴油嘴觸發
- **HSM（Hardware Security Module）**：韌體加密，防止未授權的參數修改
- **ASIL-D 合規**：ISO 26262 最高功能安全等級

### 3.2 主流車用 MCU 比較

| 製造商 | 系列名稱 | 架構特點 | 主要應用場景 |
|--------|----------|----------|-------------|
| Infineon | AURIX™ TC3xx / TC4xx | TriCore™ 多核心架構，強大 DSP 能力，支援 Lockstep | 德系車廠（Bosch、Continental）主流選擇 |
| NXP | S32K / S32G | ARM Cortex-M7 / A53，整合 CAN-FD 與車用乙太網路 | 網域控制器（Domain Controller） |
| ST Micro | Stellar / SPC5 | ARM Cortex-R52，相變記憶體（PCM）支援快速 OTA | 歐系油電混合動力總成 |
| Renesas | RH850 | 32-bit RISC 自研架構，超低功耗、極高可靠性 | 日系車廠（Toyota、Honda、Nissan）首選 |

### 3.3 研究與自製 ECU 的替代方案

對於開源或研究性質的 EFI 開發，常見的替代選擇：

- **STM32 F4 / F7 / H7 系列**：開源 ECU 計畫（如 RusEFI）的最主流選擇，社群資源豐富，最高時脈可達 480 MHz。
- **Teensy（NXP i.MX RT1062）**：600 MHz 的高效能跨界處理器，適合高階 DIY 應用。
- **Raspberry Pi RP2350**：其 **PIO（可程式化 I/O）模組** 能以零 CPU 抖動處理曲軸觸發訊號，類似專業 GTM，適合無人機（UAV）研究應用。

> **建議**：若進行自定義飛行控制器或引擎管理研究，STM32H7 或 RP2350 是兼顧效能與開發成本的優質選擇。

## 4. 關鍵感測器與輸入資訊

### 4.1 靜態硬體參數

在計算供油量前，ECU 必須先知道引擎的物理特性：

| 參數 | 說明 |
|------|------|
| 引擎排氣量 | 計算理論進氣質量的基礎（如 2000 cc） |
| 噴油嘴流量 | 通常以 cc/min 或 lbs/hr 表示，用於換算燃油脈寬 |
| 燃油壓力 | 通常假設固定壓力（如 3 bar），壓力變化時需補正表格 |
| 曲軸觸發輪型式 | ECU 需知道觸發輪的「語言」，例如 60-2 齒型代表 60 齒中缺 2 齒以識別上死點 |

### 4.2 即時感測器輸入

| 感測器 | 提供資訊 | 在 ECU 中的用途 |
|--------|----------|----------------|
| 曲軸位置感測器（CKP） | 引擎轉速（RPM）與活塞位置 | 決定點火與噴油時機 |
| 進氣岐管壓力（MAP） | 進氣壓力 | 引擎負載的主要指標（Speed-Density 演算法） |
| 進氣溫度（IAT） | 空氣溫度 | 計算空氣密度，修正供油量 |
| 冷卻水溫（CLT） | 引擎暖機狀態 | 觸發冷啟動加濃（類似手動阻風門） |
| 節氣門位置（TPS） | 駕駛者意圖 | 加速補油與引擎負載估算 |

## 5. 軟體語言與開發規範

### 5.1 主要程式語言

EFI 系統開發高度依賴底層硬體控制，代碼必須具備高度的**執行確定性（Determinism）**。

**C 語言（工業標準）**
- 允許直接操作硬體暫存器與記憶體位址
- 編譯產出的二進制程式小且執行迅速
- 生產級 ECU（Bosch、Denso）與開源計畫（Speeduino、RusEFI）均廣泛使用

**C++（現代模組化設計）**
- 物件導向設計使代碼更模組化、易於維護
- 適合用於高層「監督邏輯」，底層計時仍以 C 風格撰寫
- RusEFI 等高效能開源計畫的首選

**組合語言（Assembly）**
- 通常以「內嵌（Inline）」方式穿插於 C 代碼中
- 用於極限優化的中斷服務程序（ISR）與開機初始化程式
- 確保在 8000 RPM 下，曲軸觸發事件能在微秒內得到零開銷回應

### 5.2 模型驅動開發（Model-Based Design, MBD）

在專業車廠，工程師通常不手寫 C 代碼，而是採用 **MATLAB/Simulink** 建立數學模型：

1. 以流程圖或數學模型描述引擎行為
2. 透過自動代碼生成工具（如 TargetLink）轉換為 **MISRA C 合規**的代碼
3. MISRA C 禁止危險的編程實踐（如動態記憶體配置），確保代碼永不崩潰或進入無窮迴圈

### 5.3 各開發環境比較

| 開發環境 | 主要語言 | 最適用場景 |
|----------|----------|-----------|
| 生產級 ECU（Bosch、Denso） | C / MATLAB Simulink | 可靠性、排放法規合規、量產 |
| 開源計畫（RusEFI） | C++ | 高效能 DIY 硬體、社群協作開發 |
| 入門自製（Speeduino） | C（Arduino 框架） | 四缸引擎入門、操作簡單 |
| 研究 / 無人機（UAV） | C++ / Python | 整合強化學習（RL）或自適應控制 |

## 6. 通訊協議與安全機制

### 6.1 通訊協議

- **CAN Bus（控制器區域網路）**：用於向數位儀表板傳送轉速等資料，或接收循跡控制系統的訊號。
- **CAN-FD**：下一代 CAN 協議，支援更高的資料傳輸速率，適用於現代多控制器整合架構。
- **串口 / USB 調校協議**：允許外部電腦在不重新燒錄固件的情況下即時修改內部查找表（Maps）。

### 6.2 安全機制

- **跛行回家模式（Limp Mode）**：當偵測到感測器故障（如 MAP 感測器失效）時，ECU 自動切換至固定的安全點火圖，防止引擎損壞。
- **PID 控制迴路**：用於怠速空氣控制（IAC）和增壓控制（Boost Control），維持引擎在負載變化下的穩定性。
- **ASIL 功能安全等級**：最高的 ASIL-D 等級要求硬體能即時偵測並修正記憶體位元錯誤，進入「安全狀態」。

## 7. 結論與進階方向

EFI 編程不僅僅是撰寫程式碼，更是將**物理熱力學原理**轉化為**數位控制邏輯**的跨領域工程實踐。

對於研究領域（尤其是無人機飛行控制與小型 UAV 引擎管理），未來的發展方向包括：

- **強化學習（Reinforcement Learning）整合**：以 RL 演算法進行 PID 參數自動調適，取代手動靜態調整。
- **自適應幾何控制（Adaptive Geometric Control）**：提升系統在極端動態工況下的魯棒性與穩定性。
- **Python 輔助開發**：以 Python 進行演算法訓練與驗證，再轉換為 C++ 部署至 ECU 固件。
